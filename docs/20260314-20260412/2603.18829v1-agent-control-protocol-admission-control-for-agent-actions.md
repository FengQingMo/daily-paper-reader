## 摘要
Agent Control Protocol（ACP，代理控制协议）是一项用于在 B2B 机构环境中治理自主代理的正式技术规范。ACP 作为代理意图与系统状态变更之间的准入控制层：在任何代理动作进入执行之前，必须通过一次加密的准入检查，以同时验证身份、能力范围、委托链和策略合规性。ACP 定义了系统为使自主代理在明确的机构控制下运行所必须实现的机制，包括加密身份、基于能力的授权、确定性风险评估、可验证的链式委托、可传递撤销以及不可变审计。ACP 作为附加层运行于 RBAC 和 Zero Trust 之上，而不取代它们。v1.13 规范由 36 份技术文档组成，分为五个符合性等级（L1-L5）。它包括一个涵盖所有 L1–L4 能力的 Go 参考实现（22 个软件包）、51 个已签名的符合性测试向量（Ed25519 + SHA-256），以及针对所有 HTTP 端点的 OpenAPI 3.1.0 规范。该规范定义了超过 62 项可验证要求、12 项禁止行为，以及机构间互操作机制。规范与实现参见：https://github.com/chelof100/acp-framework-en

## Abstract
Agent Control Protocol (ACP) is a formal technical specification for governance of autonomous agents in B2B institutional environments. ACP is the admission control layer between agent intent and system state mutation: before any agent action reaches execution, it must pass a cryptographic admission check that validates identity, capability scope, delegation chain, and policy compliance simultaneously.   ACP defines the mechanisms of cryptographic identity, capability-based authorization, deterministic risk evaluation, verifiable chained delegation, transitive revocation, and immutable auditing that a system must implement for autonomous agents to operate under explicit institutional control. ACP operates as an additional layer on top of RBAC and Zero Trust, without replacing them.   The v1.13 specification comprises 36 technical documents organized into five conformance levels (L1-L5). It includes a Go reference implementation of 22 packages covering all L1-L4 capabilities, 51 signed conformance test vectors (Ed25519 + SHA-256), and an OpenAPI 3.1.0 specification for all HTTP endpoints. It defines more than 62 verifiable requirements, 12 prohibited behaviors, and the mechanisms for interoperability between institutions.   Specification and implementation: https://github.com/chelof100/acp-framework-en

## 1. 核心问题与研究动机

- **研究问题**  
  该论文关注的是在 **B2B机构环境**（如银行、政府、企业级AI系统）中，**如何对自主代理（Autonomous Agents）进行可验证的行为治理**。  
  当前人工智能代理在执行决策与行为之间缺乏中间验证层，导致：
  - 代理的行为既无法被即时干预，也缺乏可追溯审计；
  - RBAC（基于角色的访问控制）与 Zero Trust（零信任体系）虽然能限制访问权限，但无法对自主代理的 “行动” 进行实时治理和跨机构认证。  
  作者指出这是一个“结构性缺口”——在决策与执行之间需要一个正式的“准入控制层”（Admission Control Layer）。

- **研究动机与目标**  
  通过设计 **Agent Control Protocol (ACP)**，提供一个在代理执行行为前进行全面加密验证的技术框架，使得：
  - 每个自主演化的行为都需经由身份认证、能力授权、风险评估、合规审计；
  - 跨机构环境中，授权与审计可独立验证，不依赖内部信任；
  - 所有行为都具备可验证的审计记录和追溯能力。

## 2. 方法论与技术框架

### 2.1 核心思想

ACP 将人类控制体系中的 **访问控制（RBAC/Zero Trust）** 概念扩展至自主代理，形成一个 **加密准入协议层**，确保每一次代理行为都需通过多级验证：

> **代理意图 → 身份验证 → 能力授权 → 策略风险核查 → 批准或拒绝 → 行动执行 → 审计记录**

### 2.2 关键技术组件

ACP v1.13 由六大技术机制构成，每个机制均有独立规范与状态模型：

1. **ACP-SIGN-1.0：序列化与签名机制**  
   - 所有对象先规范化（JCS, RFC 8785），再进行 **SHA-256** 哈希、**Ed25519** 签名与验证；  
   - 无效签名直接拒绝，无任何例外（Fail-Closed 机制）。

2. **ACP-CT-1.0：能力令牌（Capability Token）**  
   - 定义代理在特定资源上的具体权限（capability）、有效期、可委托深度等；  
   - 含防重放随机数（nonce）与链式哈希（parent_hash）以确保可验证委托链。

3. **ACP-HP-1.0：握手与持有证明（Proof-of-Possession）**  
   - 每次请求均附带一个单次使用的挑战签名过程，确保调用者确实持有对应私钥；
   - 过程无会话状态，增强安全性与抗重放能力。

4. **ACP-RISK-1.0：确定性风险评估模型**  
   - 定义风险分数 RS ∈ [0,100]：
     - 基准风险 B(c)
     - 环境特征 Fctx(x)
     - 行为历史 Fhist(h)
     - 资源敏感度 Fres(r)  
   - 决策门限按 autonomy_level (0–4) 调整：
     - RS ≤ 39 → 批准；  
     - 40–69 → 升级人工复审；  
     - ≥ 70 → 拒绝。

5. **ACP-DCMA-1.1：可验证链式委托**  
   - 保证权限传递链条中：
     - 不得扩权（delegated ⊆ delegator）；  
     - 深度有限制；  
     - 撤销具有传递性（transitive revocation）。

6. **ACP-EXEC-1.0 与 ACP-LEDGER-1.3：执行令牌与审计账本**  
   - 执行令牌（Execution Token, ET）为一次性签名授权票据；  
   - 审计账本基于哈希链（hn = SHA-256(en || hn-1)）实现不可篡改存储；  
   - 每个事件（批准、拒绝、执行等）均被签名并可外部验证。

### 2.3 公式与逻辑定义

核心执行约束条件：

> **Execute(request) ⇒ ValidIdentity(agent) ∧ ValidCapability ∧ ValidDelegationChain ∧ AcceptableRisk**

四个条件必须同时为真，否则直接拒绝，**无例外**（PROHIB-001）。

### 2.4 信任与跨机构机制

- **ACP-ITA-1.0**：机构信任锚（Institutional Trust Anchor），记录 institution_id 与公钥绑定；  
- **ACP-ITA-1.1**：双向信任互认协议（MRA 签署），确保不同权威体系间依然可验证；  
- **ACP-CROSS-ORG-1.1**：跨机构交互模型，定义交互ID、重试协议与 ACK 事件审计，避免分布式状态不一致问题。
