## 摘要
人工智能代理在孤立的软件工程（SWE）任务中（如解决Github上的问题）已经表现出日益增强的能力。然而，涉及多个相互依赖子任务的长期任务仍在准确性和及时完成方面存在挑战。一个自然的解决途径是异步多代理协作，即多个代理同时在任务的不同部分上工作。然而，多代理系统的有效应用却出乎意料地困难：多个代理的并发编辑会相互干扰，依赖关系难以同步，而将部分进展整合为一个连贯的整体也充满挑战。另一方面，人类开发者长期以来依靠成熟的协作基础设施来应对大型软件项目中的这些挑战。受这些协作机制启发，我们提出“集中式异步隔离委派（Centralized Asynchronous Isolated Delegation，CAID）”，这是一种以三种核心SWE原语为基础的结构化多代理协调范式：集中任务委派、异步执行和隔离工作空间。CAID通过中央管理器构建具备依赖关系意识的任务计划，在隔离工作空间中并发执行子任务，并通过结构化集成及可执行的测试验证来整合进展。在实证评估中，我们发现CAID在论文复现任务（PaperBench）中比单代理基线提升了绝对26.7%的准确度，在Python库开发任务（Commit0）中提升了绝对14.3%。系统分析表明，分支与合并是多代理协作的核心协调机制，而git worktree、git commit和git merge等SWE原语使该机制能够以可靠且可执行的方式实现。

## Abstract
AI agents have become increasingly capable at isolated software engineering (SWE) tasks such as resolving issues on Github. Yet long-horizon tasks involving multiple interdependent subtasks still pose challenges both with respect to accuracy, and with respect to timely completion. A natural approach to solving these long-horizon tasks in a timely manner is asynchronous multi-agent collaboration, where multiple agents work on different parts of the task at the same time. But effective application of multi-agent systems has proven surprisingly difficult: concurrent edits by multiple agents interfere with each other, dependencies are difficult to synchronize, and combining partial progress into a coherent whole is challenging. On the other hand, human developers have long relied on mature collaboration infrastructure to manage these challenges in large software projects. Inspired by these collaboration primitives, we introduce Centralized Asynchronous Isolated Delegation (CAID), a structured multi-agent coordination paradigm grounded in three core SWE primitives: centralized task delegation, asynchronous execution, and isolated workspaces. CAID constructs dependency-aware task plans through a central manager, executes subtasks concurrently in isolated workspaces, and consolidates progress via structured integration with executable test-based verification. In empirical evaluation, we find that CAID improves accuracy over single-agent baselines by 26.7% absolute on paper reproduction tasks (PaperBench) and 14.3% on Python library development tasks (Commit0). Through systematic analysis, we find that branch-and-merge is a central coordination mechanism for multi-agent collaboration, and that SWE primitives such as git worktree, git commit, and git merge enable it to be realized in a reliable and executable manner.

## 一、核心问题与研究背景

- **研究动机**：  
  当前基于大型语言模型（LLM）的软件工程代理在处理**单一的、孤立的任务**（如修复 GitHub issue）上表现出显著提升，但在**长周期、多子任务互依赖**的复杂软件构建与论文复现任务中，仍然存在两个主要困难：  
  1. **协作干扰问题**：多个代理在同一代码库中并发修改，容易出现版本冲突或语义不一致。  
  2. **依赖同步与集成挑战**：不同子任务间的依赖难以动态追踪和同步，最终整合常导致失败。  

- **整体意义与研究目标**：  
  作者希望构建一种**结构化的异步多代理协作框架**，旨在让多个软件工程代理能够像人类开发团队一样，通过分支、隔离环境和测试验证实现可靠并行协作，从而**提高任务准确率与整体执行效率**。

## 二、方法论：CAID 框架

### 核心思想
提出 **Centralized Asynchronous Isolated Delegation (CAID)**，即“集中式异步隔离委派”框架，它将人类软件协作中的关键工程原语（git、依赖图、测试验证）映射到多智能体系统中，实现：
1. **中央管理器（Manager）**负责任务分解、依赖建模与动态调度。  
2. **工程师代理（Engineer Agents）**在各自的隔离工作区（git worktree）中并行实现和自验证。  
3. **结构化集成（Merge）**通过版本控制合并与测试门控，保证最终代码的正确性与一致性。

### 技术细节与流程

CAID 的核心操作基于六个软件工程原语：

| SWE 原语 | 在 CAID 中的作用 |
| -------- | ---------------- |
| dependency graph | 构建任务依赖有向图，确定任务的安全执行顺序 |
| git worktree | 为每个代理提供独立工作空间，实现物理隔离 |
| git commit / pull request | 用于任务完成状态汇报和迈入集成阶段 |
| git merge | 集中集成与冲突解决 |
| asyncio（异步执行） | 管理并发执行事件循环与同步机制 |
| git reset | 同步状态至最新集成版本 |

#### 算法/执行逻辑（文字化流程）
1. **任务分解与依赖建模**：管理器解析代码库或论文结构 → 构建依赖图 \(G=(V,E)\)。  
2. **任务委派**：基于依赖图筛选可执行任务 → 分配给最多 N 个并行工程师。  
3. **隔离执行**：每个工程师在独立分支开发、自测（运行相关单元测试）。  
4. **集成验证**：工程师提交 verified commit → 管理器执行 merge 与冲突解决。  
5. **循环处理与重分配**：更新依赖状态，进入下一轮任务分配直到全部完成。  

该设计避免了自由形式对话传递的误差，所有通信均使用**结构化 JSON 指令与 git 操作**。

## 六、主要结论与发现

- **性能提升显著**：  
  - PaperBench 准确率提升 26.7 个百分点（平均 57.2% → 63.3%）。  
  - Commit0-Lite 提升 14.3 个百分点（平均 53.1% → 59.1%）。  

- **核心机制验证**：  
  分支与合并（branch-and-merge）是实现稳定多代理协作的关键；  
  隔离与测试门控可显著减少并发冲突和错误集成。

- **策略洞察**：  
  - 延长单代理迭代不会显著提升性能；并行协作的收益远大于单体迭代扩展。  
  - 先运行单代理再切换多代理的“后备策略”效率低下，建议**直接采用多代理协同模式**。

## 七、优点与亮点

- **结构化设计**：将人类开发中的 git 流程（branch、merge、commit）精确映射进多代理系统，具备真实可执行性。  
- **通信可验证性**：避免自由对话，所有指令均在 JSON 格式中结构化传递。  
- **充分的实证分析**：从主实验、消融、可视化（Gantt）、统计检验多维度验证方法鲁棒性。  
- **概念创新性**：将“软件工程原语”作为多智能体协作机制的设计基元，提供了跨人机协作的工程视角。

## 八、不足与限制

- **算力与成本**：CAID 虽提升准确率，但引入更高的通信与验证开销（API 成本与运行时长均翻倍甚至三倍）。  
- **调度能力限制**：中央管理器基于**提示工程启发式**进行任务分解，不具备学习能力，导致在复杂依赖下仍可能委派不当。  
- **泛化范围有限**：目前验证仅限软件工程任务（代码库生成与论文复现），尚未验证在文档写作、设计或多模态任务中的适用性。  
- **可扩展性风险**：当代理数超过协调能力（如 N=8）时性能下降，显示任务分割与合并瓶颈。
