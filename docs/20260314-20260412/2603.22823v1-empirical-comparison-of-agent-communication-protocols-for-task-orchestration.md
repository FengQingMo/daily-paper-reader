## 摘要
背景。目前，人工智能代理系统正从单一工具交互转向复杂的多代理协作。因此，出现了两种竞争性的通信协议：一种是用于标准化代理调用外部工具的工具集成协议，另一种是使自主代理能够发现并相互委派任务的代理间委派协议。尽管已有数十家企业合作伙伴在业界广泛采用，但文献中尚不存在对这些协议的实证比较。目标。本研究旨在开发首个系统化基准，用于比较仅使用工具集成、多代理委派以及混合架构在三种复杂度的标准化查询下的表现，并量化响应时间、上下文窗口消耗、货币成本、错误恢复能力和实现复杂性之间的权衡。

## Abstract
Context. Nowadays, artificial intelligence agent systems are transforming from single-tool interactions to complex multi-agent orchestrations. As a result, two competing communication protocols have emerged: a tool integration protocol that standardizes how agents invoke external tools, and an inter-agent delegation protocol that enables autonomous agents to discover and delegate tasks to one another. Despite widespread industry adoption by dozens of enterprise partners, no empirical comparison of these protocols exists in the literature. Objective. The goal of this work is to develop the first systematic benchmark comparing tool-integration-only, multi-agent delegation, and hybrid architectures across standardized queries at three complexity levels, and to quantify the trade-offs in response time, context window consumption, monetary cost, error recovery, and implementation complexity.

## 2. 方法论与核心思想

- **总体方法**：  
  构建了一个统一的实验应用——**Open Source Health Analyzer**，以实际 API 查询任务为基准，系统对比三种通信架构：

  1. **MCP-only 架构**：单个智能体通过 MCP 连接 4 个工具服务器（GitHub、npm、OSV.dev、StackOverflow）；所有工具回复累积于同一个上下文窗口中。
  2. **A2A 多智能体架构**：一个协调器（Coordinator Agent）通过 HTTP（JSON-RPC）向 4 个专职子代理分发任务，每个代理独立维护上下文窗口。
  3. **Hybrid 架构**：在执行前用启发式路由器根据信息密度判断任务复杂度，自动选择 MCP 或 A2A 路径。

- **公式化任务定义**：  
  给定查询集合 \( Q = \{q_1, q_2, …, q_{30}\} \) 按复杂度划分为简单（simple）、中等（medium）、复杂（complex）三级。

  \[
  A^* = \arg\min_A f(P_A, c)
  \]
  其中 \( f \) 是以延迟、token 数与成本组合的加权目标函数，约束条件为可靠性=1.0（无失败及幻觉）。

- **统计分析方法**：

## 3. 实验设计

- **实验场景**：  
  应用领域为开源项目健康度分析，数据来源包括：
  - GitHub REST API（代码与仓库统计）；
  - npm 注册库（下载量）；
  - OSV.dev（安全漏洞数据）；
  - Stack Overflow（社区问答活跃度）。

- **基准构成**：  
  - 30 条自然语言查询，分为 10 简单、10 中等、10 复杂；
  - 每类查询分别在 MCP、A2A、Hybrid 架构上执行；
  - 每个查询重复 5 次，总计 **450 次运行（30×3×5）**。

- **输出指标**：  
  1. 延迟时间（秒）；  
  2. token 消耗量；  
  3. 成本（基于输入输出 token 定价）；  
  4. LLM API 调用次数；  
  5. 外部数据源 API 调用次数；  
  6. 实现代码复杂度（行数、圈复杂度）。

- **错误注入与验证**：  
  随机关闭部分数据源以测试错误恢复能力，同时构建事实验证模块，确保回答可溯源至真实 API 数据。

## 6. 主要结论与发现

- **核心发现：复杂度依赖性“性能交叉点”**  
  - 简单查询：MCP 平均延迟 **8.9 s**，快于 A2A（19.8 s），效果显著；  
  - 复杂查询：A2A 平均延迟 **45.1 s** 优于 MCP（51.8 s）；  
  - token 消耗：A2A 仅为 MCP 的 **1/3.1**；  
  - 成本差异：复杂查询中 A2A 比 MCP **节省 39% 成本**；  
  - Hybrid 架构几乎在各复杂度下追踪最佳方案。

- **定量贡献**：  
  首次测量并量化了单智能体架构“上下文窗口膨胀（context window bloat）”问题，其在复杂任务中造成约 3.1 倍的 token 负担。

- **实用决策框架**：  
  1. 简单或单源查询 → 选用 MCP；  
  2. 多源、多项目、复杂任务 → 选用 A2A；  
  3. 复杂度未知 → 使用 Hybrid 自动路由。

## 8. 不足与局限

- **模型单一性**：仅测试 Claude Sonnet，一般化至其他模型（如 GPT‑4、Gemini）仍需验证。

- **统计覆盖限制**：每查询 5 次重复能检测大效应，但不足以评估微小差异。
