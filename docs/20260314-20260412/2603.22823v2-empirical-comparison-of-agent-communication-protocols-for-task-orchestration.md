## 摘要
背景。当今，人工智能代理系统正从单一工具交互转变为复杂的多代理协作。因此，出现了两种竞争性的通信协议：一种是工具集成协议，用于标准化代理调用外部工具的方式；另一种是代理间委托协议，用于使自主代理能够相互发现并委托任务。尽管已有数十家企业合作伙伴在业界广泛采用这些协议，但目前文献中尚无针对这些协议的实证比较。目标。本研究旨在开发首个系统化基准，用以比较仅使用工具集成、多代理委托以及混合架构在三个复杂度层级的标准化查询上的表现，并量化其在响应时间、上下文窗口消耗、经济成本、错误恢复能力和实现复杂度方面的权衡。

## Abstract
Context. Nowadays, artificial intelligence agent systems are transforming from single-tool interactions to complex multi-agent orchestrations. As a result, two competing communication protocols have emerged: a tool integration protocol that standardizes how agents invoke external tools, and an inter-agent delegation protocol that enables autonomous agents to discover and delegate tasks to one another. Despite widespread industry adoption by dozens of enterprise partners, no empirical comparison of these protocols exists in the literature. Objective. The goal of this work is to develop the first systematic benchmark comparing tool-integration-only, multi-agent delegation, and hybrid architectures across standardized queries at three complexity levels, and to quantify the trade-offs in response time, context window consumption, monetary cost, error recovery, and implementation complexity.

## 一、核心问题与研究背景

- **研究动机**：随着大型语言模型（LLM）驱动的智能体系统从单工具交互向多智能体协作式编排演化，系统架构设计面临关键问题——代理之间与外部工具之间应采用何种通信机制。  
- **背景冲突**：业内出现两类主流标准：
  - **Model Context Protocol (MCP)**：由 Anthropic 提出，用于规范代理调用外部工具的方式。
  - **Agent-to-Agent Protocol (A2A)**：由 Google 提出，用于支持代理的能力发现与任务委托。  
- **研究缺口**：尽管两者被众多企业广泛使用，但目前无任何 **实证对比研究**，架构师在选择协议时只能凭经验或主观判断。  
- **目标**：建立系统化 benchmark，对 MCP、A2A 及混合架构在不同任务复杂度下的性能表现进行定量比较，揭示二者的权衡关系并提出选择框架。

## 二、方法论与技术路线

- **核心思想**：在同一应用逻辑下构建三种架构版本，并在保持语言模型一致的条件下进行公平对比，评估通信协议对整体性能的影响。
- **三种架构**：
  1. **MCP-only 架构**：单一智能体通过标准输入/输出（stdio）或 HTTP 与四个工具服务器通信，所有调用与响应在同一上下文窗口中积累。
  2. **A2A Multi-Agent 架构**：主协调代理通过 A2A 协议将任务分派给四个独立的专用子代理（HTTP + JSON-RPC），每个子代理在独立上下文内执行并返回结果。
  3. **Hybrid 架构**：在运行时通过零成本的关键字启发式路由器判断查询复杂度，针对简单任务走 MCP，复杂任务走 A2A。
- **数学形式化**：  
  对给定查询集 \( Q = \{q_1, ..., q_{30}\} \)，复杂度分级 \( c \in \{\text{simple, medium, complex}\} \)，三种架构集合 \( A \in \{\text{MCP}, \text{A2A}, \text{Hybrid}\} \)。  
  优化目标为：
  \[
  A^* = \arg \min_{A} f(P_A, c)
  \]
  其中 \( P_A = (\text{latency}, \text{tokens}, \text{cost}) \)，约束为无故障执行与输出零幻觉。  
- **统计分析**：
  - 使用 **Bootstrap 95% 置信区间 (10,000 次重采样)**；
  - **Mann–Whitney U 检验**（配合 Bonferroni 校正）；
  - **Cliff’s δ 效应量**衡量差异强度。

## 六、主要结论与发现

- **核心发现**：出现显著的性能“**复杂度交叉效应**”：
  - **简单查询**：MCP 显著更快（8.9s vs 19.8s, *p* < 0.0001, δ = −0.90）。
  - **中等查询**：MCP 仍领先（30.3s vs 36.0s）。
  - **复杂查询**：A2A 更优（45.1s vs 51.8s, δ = +0.29）。  
- **上下文窗口效应**：
  - MCP 在复杂任务中累计 34,959 tokens，较 A2A 的 11,318 多出 3.1 倍；导致成本上升 39%。  
  - “**上下文窗口膨胀 (context window bloat)**”首次被量化。  
- **成本分析**：复杂查询下 MCP 花费 $0.130，A2A $0.079。
- **混合架构表现**：在各复杂度层级上接近最优协议，设计证明其动态路由有效。
- **总体结论**：两个协议互补而非竞争——MCP 适合轻量单源查询，A2A 适合复杂任务编排，Hybrid 适合动态场景。
