## 摘要
人工智能代理生态系统目前主要采用两种协议：用于工具调用的模型上下文协议（Model Context Protocol，MCP）和用于单一主体任务委派的代理间通信协议（Agent-to-Agent，A2A）。这两种协议都假设由单一控制主体（即某个人或组织）拥有所有代理。然而，当来自独立主体的代理需要在共享状态上进行协调时（例如多个工程师的代码代理共同编辑仓库、家人协作规划旅行、或不同组织的代理协商联合决策），现有协议便无法适用，协调往往演变为临时聊天、人工合并或静默覆盖。我们提出了 MPAC（多主体代理协调协议，Multi-Principal Agent Coordination Protocol），这是一种在应用层提供显式协调语义的协议，涵盖五个层次：会话层（Session）、意图层（Intent）、操作层（Operation）、冲突层（Conflict）和治理层（Governance）。MPAC 将意图声明作为行动的前置条件，将冲突作为一类结构化对象进行表示，并通过可插拔的治理层支持人机协同仲裁。该规范定义了 21 种消息类型、三个带有规范状态转换表的状态机、基于 Lamport 时钟的因果水印、两种执行模型、三种安全配置文件，以及针对共享状态的乐观并发控制。我们发布了两个可互操作的参考实现（Python 和 TypeScript），包含 223 个测试、一个 JSON 模式套件以及七个实时多代理演示。一个包含三代理代码审查的控制性基准测试显示，与序列化的人为干预基线相比，协调开销减少 95%，整体运行速度提升 4.8 倍，且每代理决策时间保持不变。速度提升源于消除协调等待，而非压缩模型调用。协议规范、实现及演示均为开源。

## Abstract
The AI agent ecosystem has converged on two protocols: the Model Context Protocol (MCP) for tool invocation and Agent-to-Agent (A2A) for single-principal task delegation. Both assume a single controlling principal, meaning one person or organization that owns every agent. When independent principals' agents must coordinate over shared state, such as engineers' coding agents editing the same repository, family members planning a shared trip, or agents from different organizations negotiating a joint decision, neither protocol applies, and coordination collapses to ad-hoc chat, manual merging, or silent overwrites. We present MPAC (Multi-Principal Agent Coordination Protocol), an application-layer protocol that fills this gap with explicit coordination semantics across five layers: Session, Intent, Operation, Conflict, and Governance. MPAC makes intent declaration a precondition for action, represents conflicts as first-class structured objects, and supports human-in-the-loop arbitration through a pluggable governance layer. The specification defines 21 message types, three state machines with normative transition tables, Lamport-clock causal watermarking, two execution models, three security profiles, and optimistic concurrency control on shared state. We release two interoperable reference implementations in Python and TypeScript with 223 tests, a JSON Schema suite, and seven live multi-agent demos. A controlled three-agent code review benchmark shows a 95 percent reduction in coordination overhead and a 4.8 times wall-clock speedup versus a serialized human-mediated baseline, with per-agent decision time preserved. The speedup comes from eliminating coordination waits, not compressing model calls. Specification, implementations, and demos are open source.

## 一、研究动机与问题背景

- **核心问题**：现有的 AI 代理通信协议（如 MCP 与 A2A）均假设代理隶属于同一控制主体（如单个用户或组织），无法处理**多主体（multi-principal）环境**下的协作与冲突管理。  
  当不同组织、个人或系统的代理需在共享状态下共同行动（如代码协同编辑、跨组织合同撰写、多人旅行计划等），由于缺乏统一协调语义，常出现以下问题：
  - 冲突检测和解决依赖临时聊天或人工合并；
  - 缺乏行为可追溯性与责任归属；
  - 无法防止数据覆盖或语义不一致。
- **研究动机**：作者希望建立一个**应用层协调协议**，在多个独立主体间提供规范的“意图声明-冲突检测-治理仲裁”机制，使独立代理系统能安全、透明且高效地协作。
- **研究意义**：MPAC（Multi-Principal Agent Coordination Protocol）为多主体智能体系统提供缺失的协调层，与 MCP（工具调用）和 A2A（单主体任务委派）形成互补关系。

## 二、方法论与关键技术设计

### 1. 核心思想
- 在共享环境中引入统一的协调语义，要求**“先声明意图，再执行操作”**；  
- 将冲突、决策、治理等问题结构化为**协议层**的显式对象；
- 保证**可溯源、可治理、可拓展**的协作链路；
- 通过 **五层模型**（Session、Intent、Operation、Conflict、Governance）实现严格的状态与消息管理。

### 2. 五层结构设计
1. **Session 层**：确定参与方与协调器的会话管理（HELLO、HEARTBEAT、SESSION_CLOSE 等消息）。  
2. **Intent 层**：代理需在行动前声明“意图”（目标、作用范围、优先级、有效期等），便于协调器提前检测潜在冲突。  
3. **Operation 层**：执行实际状态修改，依赖乐观并发控制（OCC），通过 `state_ref_before/after` 哈希对比检测陈旧状态。  
4. **Conflict 层**：当检测到冲突时，生成**结构化冲突对象**，包括类型（scope overlap、policy violation 等）、严重性及相关代理集合。  
5. **Governance 层**：授权冲突仲裁与人工干预，确保决策的可追溯与合法性，并处理仲裁（RESOLUTION）与权限审查。

### 3. 关键技术细节
- **21 种消息类型**：覆盖五层语义，全由 JSON Schema 定义，支持跨语言验证。  
- **三类状态机**：分别管理 Intent、Operation、Conflict 生命周期（如 ACTIVE→SUSPENDED→TRANSFERRED）。  
- **Lamport 时钟因果水印**：全局事件排序，用于多代理因果追踪与审计。  
- **执行模型**：
  - *Post-commit*：先修改状态后提交，用于可信内部场景；
  - *Pre-commit*：须经协调器预审后方可执行，用于跨组织协作。  
- **安全配置**：提供 open、authenticated、verified 三种信任等级。  
- **乐观并发控制**：通过状态哈希检测冲突、支持重试与 rebase。
